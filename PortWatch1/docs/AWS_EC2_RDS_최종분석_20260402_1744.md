### 📌 스크린샷 1~2: AWS 보안그룹 목록 (Security Groups x6)

| 이름 | 역할 | 판정 |
|------|------|------|
| `default` | VPC 기본 보안그룹 | 정상 |
| `ec2-rds-1` | EC2→RDS 연결 (1번째 세트) | ✅ 정상 |
| `ec2-rds-2` | EC2→RDS 연결 (2번째 세트, RDS Proxy 시도 시 생성) | ✅ 정상 (중복이지만 문제없음) |
| `portwatch-sg` | EC2 인스턴스 기본 보안그룹 | ✅ 정상 |
| `rds-ec2-1` | RDS가 EC2 접근 허용 (1번째) | ✅ 정상 |
| `rds-ec2-2` | RDS가 EC2 접근 허용 (2번째, RDS Proxy 시도 시 생성) | ✅ 정상 (중복이지만 문제없음) |

**결론**: 보안그룹 자체는 정상. -1/-2 중복 세트는 이전 RDS Proxy 생성 시도 때 AWS가 자동 생성한 것. 삭제하지 말 것.

#### ✅ 발견 4: update-all API 성공! (모순처럼 보이지만 원인 있음)
```bash

→ {"duration":"60초","success":true,"message":"전체 주식 현재가 업데이트 완료"}
```

**왜 crawler는 실패하고 update-all은 성공?**

```
[StockScheduler] initialDelay=2분 → 매 30분마다 DB 연결 사용 중
                    ↓
[crawler/korea curl] → HikariPool에 연결 요청 → 스케줄러가 이미 사용 중 → 30초 대기 → timeout
[crawler/us curl]   → 동일 실패

[스케줄러 완료 후 연결 반환]
    ↓
[api/admin/update-all curl] → HikariPool에 빈 연결 있음 → 즉시 성공!
```

→ **스케줄러와 수동 API가 커넥션 풀을 서로 경쟁**하는 문제
